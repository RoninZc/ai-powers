# 智能体协作编排

> 本文档定义 ai-powers 框架中所有智能体的协作规则、技能加载配置、TDD 自动化编排流程、并行执行策略、失败处理机制和质量门控。

---

## 1. TDD 自动化编排流程

### 1.1 四阶段工作流概览

```
用户输入技术方案
        ↓
┌───────────────────────────┐
│ 阶段 1：需求分析与架构设计 │ (并行: architect + planner)
└───────────┬───────────────┘
            ↓ [质量门控]
┌───────────────────────────┐
│ 阶段 2：测试驱动开发       │ (循环: tdd-guide)
│ RED → GREEN → REFACTOR    │
└───────────┬───────────────┘
            ↓ [质量门控]
┌───────────────────────────┐
│ 阶段 3：质量保证           │ (并行: code-reviewer + security-reviewer + e2e-runner)
└───────────┬───────────────┘
            ↓ [质量门控]
┌───────────────────────────┐
│ 阶段 4：优化与文档         │ (并行: refactor-cleaner + doc-updater)
└───────────┬───────────────┘
            ↓
      最终编排报告
```

### 1.2 阶段 1：需求分析与架构设计

**触发条件**: 用户输入技术方案或调用 `/auto-tdd`

**并行执行**:
```
        [用户输入]
             ↓
    ┌────────┴────────┐
    ↓                 ↓
┌──────────┐    ┌──────────┐
│architect │    │ planner  │
│ 架构设计 │    │ 实施规划 │
└──────────┘    └──────────┘
    ↓                 ↓
[架构文档]       [计划文档]
    └────────┬────────┘
             ↓
      [交接文档]
```

**architect agent**:
1. 分析当前架构状态
2. 识别受影响的组件
3. 设计系统方案
4. 创建架构决策记录（ADRs）

**planner agent**:
1. 重新阐述需求
2. 分解为多个阶段
3. 创建详细的任务列表
4. 识别依赖关系和风险

**质量门控**:
- [ ] 架构设计文档包含至少 1 个 ADR
- [ ] 实施计划包含完整的阶段和任务列表
- [ ] 风险评估已覆盖关键风险
- [ ] 无 CRITICAL 架构问题

### 1.3 阶段 2：测试驱动开发（核心）

**触发条件**: 阶段 1 完成并通过质量门控

**TDD 循环**:
```
┌─────────────────────────────┐
│ 步骤 1: 定义接口和类型       │
└───────────┬─────────────────┘
            ↓
┌─────────────────────────────┐
│ 步骤 2: 编写失败的测试 (RED) │
└───────────┬─────────────────┘
            ↓
┌─────────────────────────────┐
│ 步骤 3: 运行测试 - 验证失败  │
└───────────┬─────────────────┘
            ↓
┌─────────────────────────────┐
│ 步骤 4: 实现最小代码 (GREEN) │
└───────────┬─────────────────┘
            ↓
┌─────────────────────────────┐
│ 步骤 5: 运行测试 - 验证通过  │
└───────────┬─────────────────┘
            ↓
┌─────────────────────────────┐
│ 步骤 6: 重构优化 (REFACTOR)  │
└───────────┬─────────────────┘
            ↓
┌─────────────────────────────┐
│ 步骤 7: 验证覆盖率 ≥ 80%     │
└───────────┬─────────────────┘
            ↓
      下一个任务
```

**质量门控**:
- [ ] 测试覆盖率 ≥ 80%
- [ ] 所有测试通过
- [ ] 严格遵循 TDD 循环
- [ ] 无 CRITICAL 代码质量问题

**并行策略**:
- 独立模块可并行开发
- 最多同时运行 `MAX_PARALLEL_AGENTS` 个实例（默认 5）

### 1.4 阶段 3：质量保证（并行）

**触发条件**: 阶段 2 所有任务完成

**并行执行**:
```
    [阶段 2 完成]
          ↓
┌─────────┼─────────┐
↓         ↓         ↓
┌────┐  ┌────┐  ┌────┐
│code│  │secu│  │e2e │
│revi│  │rity│  │run │
│ewer│  │revi│  │ner │
└────┘  │ewer│  └────┘
  ↓     └────┘     ↓
  ↓        ↓       ↓
  └────────┴───────┘
           ↓
    [质量保证报告]
```

**审查清单**:

**安全性（CRITICAL）**:
- [ ] 硬编码凭据
- [ ] SQL 注入
- [ ] XSS 漏洞
- [ ] 路径遍历
- [ ] CSRF 漏洞

**代码质量（HIGH）**:
- [ ] 超大函数（>50 行）
- [ ] 过深嵌套（>4 层）
- [ ] 缺少错误处理
- [ ] console.log 语句

**质量门控**:
- [ ] 无 CRITICAL 问题
- [ ] HIGH 问题 ≤ 3 个
- [ ] 所有 E2E 测试通过
- [ ] 安全审查通过

### 1.5 阶段 4：优化与文档（并行）

**触发条件**: 阶段 3 完成

**并行执行**:
```
    [阶段 3 完成]
          ↓
    ┌─────┴─────┐
    ↓           ↓
┌────────┐  ┌────────┐
│refactor│  │  doc   │
│cleaner │  │updater │
└────────┘  └────────┘
    ↓           ↓
    └─────┬─────┘
          ↓
    [最终报告]
```

**refactor-cleaner**:
1. 检测死代码
2. 检测重复项
3. 安全移除
4. 整合重复项

**doc-updater**:
1. 生成代码映射图
2. 更新文档
3. 提取 API 文档

**最终验证**:
```bash
npm run build          # 构建检查
npm run test           # 所有测试
npm run test:coverage  # 覆盖率检查
npm run lint           # 代码检查
npm run e2e            # E2E 测试
```

## 2. 并行执行策略

### 2.1 并行执行原则

1. **独立任务并行** - 无依赖关系的任务同时执行
2. **相同角色不并行** - 同一 agent 不并行处理多个任务
3. **优先级队列** - CRITICAL > HIGH > MEDIUM > LOW

### 2.2 并行执行示例

**阶段 1 并行**:
```
任务: 实现用户认证系统

并行启动:
├─ architect (3分钟)
└─ planner (4分钟)

总时间: max(3, 4) = 4 分钟
顺序执行: 3 + 4 = 7 分钟
节省: 43%
```

**阶段 3 并行**:
```
并行启动:
├─ code-reviewer (5分钟)
├─ security-reviewer (3分钟)
└─ e2e-runner (8分钟)

总时间: max(5, 3, 8) = 8 分钟
顺序执行: 5 + 3 + 8 = 16 分钟
节省: 50%
```

## 3. 失败处理机制

### 3.1 失败处理状态机

```
失败检测
    ↓
┌───┴───┐
↓       ↓
可修复  需介入
    ↓       ↓
自动修复  通知用户
    ↓       ↓
重新验证 ←──┘
    ↓
成功/失败
```

### 3.2 各类型失败处理

#### 构建失败

| 错误类型 | 可自动修复 | 修复方法 | 重试次数 |
|----------|-----------|----------|----------|
| 类型错误 | ✅ | 添加类型标注 | 3 |
| 导入错误 | ✅ | 修复导入路径 | 3 |
| 缺失依赖 | ✅ | 安装包 | 1 |
| 配置错误 | ✅ | 修复配置文件 | 2 |

**处理流程**:
1. build-error-resolver 自动介入
2. 应用最小化修复
3. 重新构建
4. 如果 3 次失败，通知用户

#### 测试失败

| 失败类型 | 处理方法 |
|----------|----------|
| 测试错误 | 修复测试 |
| 实现错误 | 修复实现 |
| 依赖问题 | 修复依赖 |
| 超时 | 增加超时或优化 |

**处理流程**:
1. 分析失败原因
2. 回到 tdd-guide 修复
3. 重新运行测试
4. 如果 3 次失败，通知用户

#### 代码审查失败

| 级别 | 问题数 | 行动 | 阻断 |
|------|--------|------|------|
| CRITICAL | > 0 | 必须修复 | ✅ |
| HIGH | > 3 | 阻断 | ✅ |
| HIGH | 0-3 | 警告 | ❌ |
| MEDIUM/LOW | 任意 | 记录 | ❌ |

#### 安全审查失败

| 级别 | 问题数 | 行动 | 阻断 |
|------|--------|------|------|
| CRITICAL | > 0 | 立即阻断 | ✅ |
| HIGH | > 0 | 建议 | ❌ |

### 3.3 用户介入条件

以下情况需要用户介入：
1. CRITICAL 问题超过 3 个
2. 同一问题 3 次自动修复失败
3. 需要业务逻辑决策
4. 需要架构修改
5. 发现 CRITICAL 安全问题

## 4. 质量门控定义

### 4.1 质量门控矩阵

| 阶段 | 门控条件 | 阻断条件 |
|------|----------|----------|
| 阶段 1 | 架构设计完整<br>实施计划可执行<br>风险评估全面 | 架构设计不完整<br>实施计划不可执行 |
| 阶段 2 | 覆盖率 ≥ 80%<br>所有测试通过<br>无 CRITICAL 问题 | 覆盖率 < 80%<br>测试失败<br>CRITICAL 问题 |
| 阶段 3 | 无 CRITICAL 问题<br>HIGH 问题 ≤ 3<br>所有 E2E 测试通过 | CRITICAL 问题<br>HIGH 问题 > 3<br>E2E 测试失败 |
| 阶段 4 | 所有测试通过<br>文档已更新 | 测试失败 |

### 4.2 测试覆盖率门控

**最低要求**: 80%

**特殊要求**（100% 覆盖率）:
- 金融计算
- 认证逻辑
- 安全关键代码
- 核心业务逻辑

**覆盖率检查**:
```bash
npm run test:coverage

# 验证覆盖率 ≥ 80%
```

### 4.3 代码审查门控

**批准标准**:
- CRITICAL 问题 = 0 → 批准
- HIGH 问题 ≤ 3 → 警告
- HIGH 问题 > 3 → 阻断

**阻断条件**:
- 任何 CRITICAL 问题
- HIGH 问题 > 3 个

### 4.4 安全审查门控

**必须标记的问题**（CRITICAL）:
- 硬编码凭据
- SQL 注入
- XSS 漏洞
- 路径遍历
- 身份验证绕过

**安全检查命令**:
```bash
npm audit --audit-level=high
npx eslint . --plugin security
```
